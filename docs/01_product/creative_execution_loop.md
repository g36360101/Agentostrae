# 创作执行循环：从文档规划到产品内 Loop

## 1. 核心判断

原始 172 份文档最有价值的部分，不是每一个模块本身，而是它们共同指向了一种产品形态：

> AgentOS 不只是一个生成器，而是一个可以陪作者反复推进作品设定的创作执行循环。

这个循环必须满足四个条件：

1. 每一步都有明确输入。
2. 每一步都有结构化输出。
3. 每一步都能保存状态。
4. 每一步都允许用户确认、暂停、回退或进入下一轮。

## 2. Loop 的产品定义

AgentOS 的创作执行循环不是后台自动跑完的流水线，而是一个由用户掌控的半自动工作流。

```text
用户输入
  -> AI 生成建议
  -> 系统结构化解析
  -> 用户审核确认
  -> 写入项目资产
  -> 构建下一步上下文
  -> 进入下一轮
```

关键点：

- AI 负责扩展和整理。
- 系统负责保存和校验。
- 用户负责判断和确认。

## 3. 第一版主循环

MVP 只实现一个主循环：

```text
Idea
  -> HighConcept
  -> CoreCard
  -> DevelopmentPlan
  -> SuggestedAssets
  -> SuggestedRelations
  -> ConfirmedKnowledgeBase
  -> ContextPack
  -> ProjectChat
  -> Export
```

这个循环的目标是让用户完成一次作品设定生成与沉淀。

## 4. Loop 节点说明

### 4.1 Idea：创意输入

用户输入一句话、一段设定、一个脑洞或一个类型方向。

输入示例：

```text
一个失忆修仙者发现自己其实是天道写废的草稿。
```

输出：

- ProjectIdea
- AIJob: generate_high_concepts

### 4.2 HighConcept：高概念候选

AI 生成 3 到 5 个候选方向。

每个方向包含：

- 标题。
- 一句话卖点。
- 主冲突。
- 目标读者。
- 情绪承诺。
- 风险提示。

用户动作：

- 选择一个。
- 要求重生成。
- 混合多个候选。

### 4.3 CoreCard：作品核心卡

把被选中的方向压缩成作品的“正史起点”。

包含：

- 作品名。
- 类型。
- 一句话卖点。
- 世界观摘要。
- 主角摘要。
- 核心矛盾。
- 长线悬念。
- 目标读者。

用户动作：

- 编辑。
- 保存。
- 重新生成局部字段。

### 4.4 DevelopmentPlan：作品开发案

基于核心卡生成更完整的创作蓝图。

包含：

- 世界背景。
- 角色组。
- 阵营组。
- 世界规则。
- 主要冲突。
- 分卷 / 阶段草案。
- 爽点与情绪曲线。
- 风险与待确认项。

用户动作：

- 接受。
- 编辑。
- 要求展开某部分。
- 进入资产抽取。

### 4.5 SuggestedAssets：候选资产

AI 从开发案中抽取结构化设定。

资产类型：

- character
- location
- organization
- item
- rule
- event
- mystery
- foreshadowing
- theme

用户动作：

- 批量确认。
- 单个编辑。
- 拒绝。
- 合并重复资产。

### 4.6 SuggestedRelations：候选关系

AI 从开发案和资产中抽取关系。

关系类型：

- belongs_to
- allies_with
- opposes
- knows
- hides_from
- caused_by
- depends_on
- located_in
- foreshadows
- reveals

用户动作：

- 确认关系。
- 修改关系类型。
- 删除不可靠关系。

### 4.7 ConfirmedKnowledgeBase：确认后的设定库

只有用户确认后的资产和关系才进入项目知识库。

知识库是后续所有 AI 对话的基础。

### 4.8 ContextPack：上下文包

系统根据用户当前任务，从知识库里构建 AI 可用上下文。

包含：

- 项目摘要。
- 关键资产。
- 关键关系。
- 隐藏真相。
- 禁止泄露项。
- 本轮任务目标。

### 4.9 ProjectChat：项目内 AI 对话

AI 基于 context_pack 回答，而不是从零开始。

回答可以产生：

- 新建议。
- 新资产候选。
- 冲突提醒。
- 继续生成任务。

### 4.10 Export：导出

用户可以导出当前阶段成果。

第一版导出 Markdown：

- 项目简介。
- 核心卡。
- 开发案。
- 资产库。
- 关系清单。
- 上下文摘要。

## 5. 第二版质量循环

主循环跑通后加入质量循环：

```text
ConfirmedKnowledgeBase
  -> QualityCheck
  -> RiskDiagnostics
  -> RepairSuggestions
  -> UserReview
  -> KnowledgeBaseUpdate
```

质量检查包括：

- 设定冲突。
- 角色动机缺失。
- 世界规则不一致。
- 伏笔无回收计划。
- 隐藏真相泄露风险。
- 上下文过长。

## 6. 第三版创作循环

质量循环稳定后，再进入创作生产循环：

```text
KnowledgeBase
  -> VolumePlan
  -> ChapterTasks
  -> TrialWriting
  -> DraftQualityReport
  -> RepairPlan
  -> KnowledgeAssetUpdate
```

这部分才接近 StoryOS，不属于第一版。

## 7. Loop 的暂停与恢复

每个 loop 节点都必须能保存：

- 当前状态。
- 输入。
- 输出。
- 用户选择。
- AIJob 记录。
- 下一步建议。

用户下次回来时，系统应该知道：

```text
你上次已经生成了开发案，但还没有确认设定资产。
```

而不是让用户重新开始。

## 8. Loop 的产品界面建议

项目工作台应该显示一个清晰的进度条：

```text
创意 -> 高概念 -> 核心卡 -> 开发案 -> 资产审核 -> 关系审核 -> 上下文包 -> 对话 -> 导出
```

每个节点显示：

- 未开始。
- 进行中。
- 待审核。
- 已完成。
- 需要修复。

## 9. 产品原则

### 9.1 不做黑箱自动化

不要设计成“一键生成完整作品设定”。

更好的体验是：

```text
AI 推进一步，用户确认一步。
```

### 9.2 不让文本悬空

AI 生成的每段关键内容，都应该能转成可保存对象。

### 9.3 不把远期治理提前带入第一版

原始文档里关于联邦治理、商业信任治理、运行时干预的内容，最多作为未来参考，不进入 MVP loop。

### 9.4 Loop 的价值是积累

每跑一轮，项目应该变得更清晰：

- 资产更多。
- 关系更明确。
- 上下文更稳定。
- AI 回答更贴合项目。

## 10. 第一版成功标准

如果用户可以完成下面的流程，就说明 loop 成立：

1. 输入一个创意。
2. 生成多个方向。
3. 选择方向。
4. 生成核心卡。
5. 生成开发案。
6. 抽取并确认至少 10 个资产。
7. 抽取并确认至少 5 条关系。
8. 构建上下文包。
9. 基于上下文继续提问。
10. 导出一份作品设定文档。
