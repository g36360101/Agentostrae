# 初期、中期、后期开发计划

## 1. 总体判断

AgentOS 当前最适合按三段推进：

```text
初期：做出可用闭环
中期：做成可迭代的创作工作台
后期：扩展为 StoryOS 创作操作系统
```

第一阶段不要追求宏大平台能力，先让作者能真实完成一次作品设定流程。中期再解决管理、质量、版本和上下文稳定性。后期才考虑试写、读者模拟、工作流控制台、多项目知识资产和治理能力。

## 2. 初期计划：0 到 8 周

### 2.1 阶段目标

做出一个可以被作者实际使用的作品设定 MVP。

用户应该能完成：

```text
创建项目
  -> 输入创意
  -> 生成高概念候选
  -> 选择方向
  -> 生成作品核心卡
  -> 生成作品开发案
  -> 抽取资产和关系
  -> 人工确认入库
  -> 构建上下文包
  -> 项目内 AI 对话
  -> 导出 Markdown 作品资料
```

初期成功的标准不是功能多，而是这条链路稳定、清楚、可重复。

### 2.2 产品范围

初期只做 P0 主链路：

- 项目系统。
- 创意输入。
- 高概念生成。
- 作品核心卡。
- 作品开发案。
- 设定资产库。
- 基础关系。
- AI 上下文包。
- 项目内 AI 对话。
- Markdown 导出。

初期不做：

- 复杂关系图谱编辑器。
- 自动长篇正文生成。
- 读者模拟。
- 连载数据分析。
- 多 Agent 会议。
- Agent 信誉分。
- 企业治理。

### 2.3 推荐节奏

| 时间 | 主题 | 关键结果 |
|---|---|---|
| 第 1 周 | 工程地基 | 前端、后端、数据库、共享类型、AI 核心包可启动 |
| 第 2 周 | 项目和创意 | 用户能创建项目、输入创意、看到项目工作台 |
| 第 3 周 | 高概念和核心卡 | AI 或 Mock 能生成高概念，用户能保存核心卡 |
| 第 4 周 | 开发案 | 基于核心卡生成结构化开发案 |
| 第 5 周 | 资产抽取 | 从开发案抽取人物、地点、组织、规则、事件、谜团、伏笔 |
| 第 6 周 | 关系和审核 | 抽取基础关系，用户能确认、拒绝、编辑候选项 |
| 第 7 周 | 上下文和对话 | 构建 context_pack，项目内 AI 对话能引用已确认设定 |
| 第 8 周 | 导出和验收 | 导出 Markdown，跑通 3 个样例项目，修复阻断问题 |

### 2.4 初期交付物

产品交付：

- 项目列表页。
- 创建项目页。
- 项目工作台。
- 创意输入区。
- 高概念候选区。
- 作品核心卡页。
- 作品开发案页。
- 资产审核页。
- 资产库页。
- 基础关系列表。
- AI 对话页。
- Markdown 导出页。

工程交付：

- `apps/web` 前端工程。
- `apps/api` 后端工程。
- `packages/shared` 类型和校验规则。
- `packages/db` 数据模型和迁移。
- `packages/ai-core` AI 任务、Prompt 模板和 Mock 输出。
- `.env.example`。
- 本地启动说明。

内容交付：

- 3 个样例项目。
- 高概念 Mock 输出。
- 核心卡 Mock 输出。
- 开发案 Mock 输出。
- 资产和关系抽取 Mock 输出。
- Markdown 导出模板。

### 2.5 初期验收标准

必须满足：

- 新开发者 30 分钟内能本地启动。
- 用户可以创建并打开一个项目。
- 一个创意能生成至少 3 个高概念候选。
- 每个高概念包含卖点、主冲突、主角驱动力、情绪承诺、风险。
- 核心卡包含读者承诺、主角缺口、对立力量、不可破坏设定。
- 开发案可以抽取至少 10 个有效资产。
- 资产有叙事功能、剧透等级和来源证据。
- 关系有证据文本、关系类型和剧透标记。
- AI 对话能引用已确认资产，而不是乱编。
- Markdown 导出像一份作者能使用的作品资料。

### 2.6 初期风险

主要风险：

- 过早追求平台化，导致主链路迟迟跑不通。
- AI 输出字段不稳定，后续很难入库。
- Mock 内容太薄，界面看似完成但无法验证真实体验。
- 资产和关系没有人工确认，污染作品正史。

应对方式：

- 每周都要有可演示用户动作。
- 所有 AI 输出先按结构化 schema 设计。
- 先用高质量 Mock 验证内容深度。
- 所有 AI 建议默认进入 suggested 状态。

## 3. 中期计划：2 到 6 个月

### 3.1 阶段目标

把工具从“能生成设定”升级为“能管理、迭代、修复设定”的创作工作台。

中期的关键词是：

```text
版本、质量、图谱、上下文、沉淀、协作效率
```

用户不仅能生成第一版设定，还能反复修改、检查冲突、管理伏笔、维护隐藏真相，并把创作判断沉淀下来。

### 3.2 产品增强

优先做：

- 关系图谱可视化 v1。
- 资产详情页增强。
- 资产版本历史。
- 设定变更记录。
- 隐藏真相 / 剧透保护。
- 伏笔与揭示计划。
- 分卷规划。
- 章节任务草案。
- 沉淀池 v1。
- DOCX / PDF 导出。

可以评估后做：

- 多平台改写提示词包。
- 角色成长弧管理。
- 世界规则一致性面板。
- 项目级搜索。
- 资产标签体系。

### 3.3 AI 增强

优先做：

- context_pack 质量评分。
- token 预算和上下文裁剪。
- 隐藏内容阻断。
- 资产冲突检测。
- 关系冲突检测。
- 开发案质量检查。
- 伏笔未回收提醒。
- AI 建议资产入库前审核增强。
- AI 回答引用来源显示。

中期不要急着做多 Agent。先让单 Agent 的上下文、引用、审核和修复能力稳定。

### 3.4 工程增强

优先做：

- 后台任务队列。
- AI 调用日志。
- 成本统计。
- API 权限和项目权限。
- 统一错误处理。
- E2E 测试。
- 数据备份策略。
- 导出任务异步化。

需要保持克制：

- 不要为了远期治理提前引入复杂事件总线。
- 不要过早拆成过多微服务。
- 不要在没有真实使用数据前做复杂推荐系统。

### 3.5 中期交付物

产品交付：

- 图谱视图。
- 版本历史。
- 伏笔管理。
- 隐藏真相管理。
- 冲突检测报告。
- 沉淀池页面。
- 分卷规划页面。
- DOCX / PDF 导出。

工程交付：

- 后台任务系统。
- AI Job 可观测面板。
- E2E 验收用例。
- 导出服务。
- 成本统计。
- 权限模型。

内容交付：

- 质量检查模板。
- 伏笔回收模板。
- 分卷规划模板。
- 沉淀池 Markdown 模板。
- 样例项目的完整中期数据。

### 3.6 中期验收标准

必须满足：

- 用户可以查看资产变更历史。
- 用户可以知道某个设定从哪里来、何时被确认、被谁修改。
- 图谱能帮助用户理解人物、组织、地点、伏笔之间的关系。
- AI 对话能明确引用资产来源。
- 系统能识别明显设定冲突。
- context_pack 能在 token 限制内保留关键设定。
- 隐藏真相不会被普通导出和普通对话误泄露。
- 用户可以导出更完整的 DOCX / PDF 资料。

### 3.7 中期风险

主要风险：

- 图谱做得太复杂，拖慢核心体验。
- 质量检测变成机械打分，没有创作价值。
- 上下文裁剪过度，导致 AI 回答丢失关键设定。
- 版本历史和沉淀池变成没人看的日志。

应对方式：

- 图谱 v1 只做查看和基础筛选，不做复杂编辑。
- 质量检查必须给出可行动建议。
- context_pack 每次构建都显示包含和排除的内容。
- 沉淀池优先记录关键创作判断，而不是所有系统事件。

## 4. 后期计划：6 到 18 个月

### 4.1 阶段目标

从作品设定工具扩展为 StoryOS 创作操作系统。

后期的目标不是“生成更多文本”，而是把创作从设定阶段延伸到试写、反馈、修复、连载管理和跨项目知识复用。

### 4.2 创作生产链

后期可以扩展：

- 试写启动器。
- 章节任务生成。
- 草稿质量报告。
- 修复计划。
- Before / After 对比。
- 章节级设定引用检查。
- 连载看板。
- 真实读者反馈导入。
- 反馈到设定库的回流机制。

重点是让作品从设定资产进入持续生产，而不是一键生成正文。

### 4.3 智能协作层

可以逐步探索：

- 图驱动 Agent。
- 工作流控制台。
- Checkpoint / Rollback。
- 策略实验。
- 创作经验图谱。
- 项目知识资产库。
- 多项目经验复用。

多 Agent 能力只有在这些基础稳定后再做：

- 资产来源清楚。
- 用户审核链路稳定。
- 上下文包可控。
- 质量检查能落地。
- 用户确实需要多个视角协作。

### 4.4 组织级能力

如果产品进入团队或商业化创作场景，可以考虑：

- 团队协作。
- 项目权限。
- 审计记录。
- 模板市场。
- 多项目资产复用。
- 工作流模板。
- 创作策略库。
- 组织级知识库。

这些能力属于后期，不应该成为 MVP 前置条件。

### 4.5 后期交付物

产品交付：

- 试写工作台。
- 章节任务系统。
- 草稿质量报告。
- 修复计划页面。
- 连载管理看板。
- 真实反馈导入。
- 工作流控制台。
- 创作经验图谱。

工程交付：

- 更完整的工作流引擎。
- Checkpoint / Rollback。
- 多项目知识检索。
- 长任务调度。
- 更细的权限和审计。
- 插件化 Prompt / Workflow 模板。

内容交付：

- 章节任务模板。
- 草稿质量评分模板。
- 修复计划模板。
- 读者反馈归因模板。
- 创作经验复用模板。

### 4.6 后期验收标准

后期成立的标志：

- 用户能从设定进入章节试写。
- AI 能指出草稿与设定库的冲突。
- 系统能把真实反馈转化为可处理的修改建议。
- 用户能管理一个连载项目的长期创作节奏。
- 多个项目之间可以复用创作经验，而不是复制粘贴旧文档。
- 工作流可以暂停、恢复、回滚。

### 4.7 后期风险

主要风险：

- 重新膨胀成过大的 StoryOS 愿景，失去核心用户体验。
- 多 Agent 和治理能力先于真实需求出现。
- 生成正文质量无法稳定，反而稀释设定工具价值。
- 团队协作、模板、权限、审计等复杂度拖垮产品节奏。

应对方式：

- 每个后期能力都必须回到真实创作场景验证。
- 先做单用户强体验，再做团队协作。
- 正文生产能力从“试写和修复”开始，不从“自动写完整本书”开始。
- 治理能力从真实协作风险中长出来，而不是从规划里先长满。

## 5. 阶段推进原则

### 5.1 每个阶段只解决一个核心问题

- 初期：能不能跑通设定闭环。
- 中期：能不能持续管理和迭代设定。
- 后期：能不能支撑完整创作生产。

### 5.2 每个阶段都要有可演示结果

不要用“底层能力完成”作为唯一交付。

每个阶段都必须能演示：

- 用户做了什么。
- 系统生成了什么。
- 用户确认了什么。
- 数据沉淀到了哪里。
- 下一步如何继续。

### 5.3 AI 永远不能绕过用户确认

无论进入哪个阶段，都保持这条原则：

```text
AI 建议 -> 系统结构化 -> 用户审核 -> 确认入库
```

这是 AgentOS 和普通 AI 聊天工具的关键区别。

### 5.4 内容质量先于功能数量

如果高概念、核心卡、开发案、资产和关系本身很薄，功能再多也没有价值。

每个阶段都要保留内容质量验收：

- 是否有叙事动力。
- 是否有来源证据。
- 是否有冲突结构。
- 是否有伏笔和揭示计划。
- 是否能支持后续 AI 协作。

## 6. 推荐里程碑

| 里程碑 | 时间 | 标志 |
|---|---:|---|
| M0 | 第 1 周 | 工程可启动，Mock schema 明确 |
| M1 | 第 3 周 | 创意到核心卡跑通 |
| M2 | 第 6 周 | 开发案到资产审核跑通 |
| M3 | 第 8 周 | 上下文、对话、导出跑通，MVP 可试用 |
| M4 | 第 3 个月 | 图谱、版本、冲突检查和 DOCX 导出可用 |
| M5 | 第 6 个月 | 沉淀池、伏笔管理、分卷规划稳定 |
| M6 | 第 12 个月 | 试写、反馈、修复和连载管理形成闭环 |

## 7. 一句话路线

初期把“作品设定闭环”做出来；中期把“设定管理和质量迭代”做扎实；后期再把它扩展成能支撑试写、反馈、修复和长期连载的 StoryOS。
