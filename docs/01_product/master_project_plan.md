# AgentOS 作品设定工具：项目经理版开发计划

## 1. 项目定位

AgentOS 当前阶段要做的不是一个庞大的 AI 创作帝国，而是一个能让作者立刻获得价值的「作品设定工具」。

它的第一版应该解决一个具体问题：

> 作者有一个模糊创意，但缺少稳定、结构化、可复用的作品设定系统。AgentOS 帮作者把创意整理成作品核心卡、开发案、设定资产、关系草图和 AI 可引用的上下文包。

第一版成功的标志不是功能多，而是这条链路能跑通：

```text
创建项目 -> 输入创意 -> 生成高概念 -> 选择方向 -> 生成核心卡 -> 生成开发案 -> 抽取资产 -> 用户确认 -> 构建上下文包 -> AI 对话 -> 导出
```

## 2. 项目管理原则

### 2.1 先闭环，后扩展

任何功能如果不能帮助 MVP 主链路闭环，就先不做。

暂时不做：

- 复杂图谱编辑器。
- 自动长篇正文生成。
- 多 Agent 会议。
- Agent 信誉分。
- 元学习。
- 联邦治理。
- 商业信任治理。

### 2.2 先协议，后智能

AI 能力可以先 Mock，但数据结构、接口协议、用户流程必须一开始就认真设计。

原因很简单：模型可以换，Prompt 可以改，但糟糕的数据结构会长期拖累产品。

### 2.3 AI 只建议，用户做最终确认

AI 可以生成资产、关系、开发案和修复建议，但不能直接污染项目正史。

所有 AI 建议至少要经历：

```text
suggested -> confirmed / rejected
```

### 2.4 每周必须有可演示结果

项目早期最怕写很多“基础设施”，但用户看不到任何东西。

每周交付物必须包含至少一个可演示用户动作，例如：

- 创建一个项目。
- 生成一组高概念。
- 保存一张作品核心卡。
- 确认一批设定资产。
- 导出一份 Markdown 设定文档。

## 3. 总体阶段划分

| 阶段 | 时间 | 目标 | 关键产出 |
|---|---:|---|---|
| Phase 0 | 1 周 | 项目启动和技术地基 | 仓库、工程骨架、数据模型、Mock 方案 |
| Phase 1 | 2 周 | 创意到作品核心卡 | 项目系统、高概念生成、核心卡 |
| Phase 2 | 2 周 | 开发案与资产库 | 开发案生成、资产抽取、关系抽取、人工确认 |
| Phase 3 | 2 周 | 上下文包与 AI 对话 | context_pack、项目内对话、上下文引用 |
| Phase 4 | 1 周 | 导出与 MVP 验收 | Markdown 导出、样例测试、问题修复 |
| Phase 5 | 4-8 周 | 产品增强 | 图谱、版本、冲突检测、DOCX 导出 |
| Phase 6 | 3-12 月 | StoryOS 扩展 | 试写、读者模拟、反馈回流、工作流控制台 |

## 4. Phase 0：项目启动和技术地基

### 目标

让团队或 AI Coding 可以稳定进入开发，而不是继续停留在文档规划。

### 主要任务

1. 初始化前端工程。
2. 初始化后端工程。
3. 建立共享类型包。
4. 建立数据库包。
5. 建立 AI 核心包。
6. 定义最小数据模型。
7. 设计 API 返回格式。
8. 准备本地运行说明。
9. 准备 Mock AI 输出样例。

### 推荐技术栈

- 前端：React + TypeScript，优先 Next.js 或 Vite。
- 后端：NestJS + TypeScript。
- 数据库：PostgreSQL。
- ORM：Prisma。
- API 数据校验：Zod 或 class-validator。
- AI 调用层：统一 AIOrchestratorService。
- 前端请求：React Query。

### 交付物

- `apps/web` 可启动。
- `apps/api` 可启动。
- `packages/shared` 有基础类型。
- `packages/db` 有 Prisma schema 草案。
- `packages/ai-core` 有 AI task 类型和 Mock 输出。
- `.env.example`。
- 本地启动说明。

### 验收标准

- 新开发者可以根据 README 在 30 分钟内跑起项目。
- 后端有健康检查接口。
- 前端能打开一个空工作台页面。
- 数据库可以迁移成功。
- Mock AI 输出格式可被后端解析。

## 5. Phase 1：创意到作品核心卡

### 目标

让用户从一句创意得到多个可选作品方向，并保存一张作品核心卡。

### 用户流程

```text
创建项目 -> 输入创意 -> 生成 3-5 个高概念 -> 选择一个方向 -> 生成作品核心卡 -> 保存
```

### 主要任务

1. 项目列表页。
2. 创建项目表单。
3. 项目工作台首页。
4. 创意输入模块。
5. 高概念生成 API。
6. 高概念候选展示。
7. 选择高概念。
8. 作品核心卡生成 API。
9. 作品核心卡展示和编辑。
10. 保存核心卡。

### 数据对象

- Project
- ProjectIdea
- HighConceptCandidate
- ProjectCoreCard
- AIJob

### 交付物

- 一个用户可以创建项目。
- 一个用户可以输入创意。
- 系统可以生成或 Mock 生成多个高概念。
- 用户可以选择一个高概念。
- 系统可以生成作品核心卡。
- 用户可以保存并再次查看作品核心卡。

### 验收标准

用一个样例创意测试：

```text
一个失忆的修仙者发现自己其实是天道写废的草稿。
```

必须能生成：

- 至少 3 个高概念候选。
- 每个候选有标题、卖点、冲突、目标读者。
- 一张结构完整的作品核心卡。

## 6. Phase 2：开发案与设定资产库

### 目标

把作品核心卡扩展为完整开发案，并从开发案中抽取可管理的设定资产。

### 用户流程

```text
核心卡 -> 生成开发案 -> 抽取资产与关系 -> 用户审核 -> 确认入库
```

### 主要任务

1. 生成作品开发案。
2. 展示开发案 Markdown。
3. 支持用户编辑开发案。
4. 从开发案抽取 suggested_assets。
5. 从开发案抽取 suggested_relations。
6. 资产审核界面。
7. 关系审核界面。
8. 确认后写入资产库。
9. 拒绝不合适资产。
10. 资产列表和资产详情页。

### 数据对象

- DevelopmentPlan
- StoryAsset
- StoryRelation
- AIJob

### 资产类型

第一版只保留：

- character
- location
- organization
- item
- rule
- event
- mystery
- foreshadowing
- theme

### 交付物

- 作品开发案生成。
- 资产抽取结果。
- 关系抽取结果。
- 人工确认流程。
- 已确认资产库。

### 验收标准

从一份开发案中至少能抽取：

- 3 个人物。
- 2 个地点或组织。
- 2 条世界规则。
- 2 个谜团或伏笔。
- 5 条基础关系。

所有资产必须有状态：

```text
suggested / confirmed / rejected / archived
```

## 7. Phase 3：上下文包与项目内 AI 对话

### 目标

让 AI 不再只基于用户当前输入回答，而是能基于项目设定资产回答。

### 用户流程

```text
选择资产范围 -> 构建 context_pack -> 项目内提问 -> AI 基于上下文回答 -> 提出新建议
```

### 主要任务

1. context_pack 数据结构。
2. 默认 context_pack 构建。
3. 基于资产类型筛选上下文。
4. 上下文摘要生成。
5. 项目 AI 对话 API。
6. 对话页面。
7. 回答中标记引用了哪些资产。
8. AI 回答中允许提出 suggested_assets。

### 数据对象

- ContextPack
- AIJob
- ChatSession
- ChatMessage
- StoryAsset
- StoryRelation

### 交付物

- 用户可以构建一个上下文包。
- 用户可以基于上下文包和 AI 对话。
- AI 回答能引用项目设定。
- AI 可以提出新的资产建议，但不能自动确认。

### 验收标准

用户问：

```text
请基于当前设定，帮我设计第一卷的主要冲突。
```

AI 回答必须体现：

- 主角设定。
- 核心矛盾。
- 至少 3 个已确认资产。
- 不乱编与设定冲突的信息。

## 8. Phase 4：导出与 MVP 验收

### 目标

让用户能把项目成果带走，而不是只留在系统里。

### 主要任务

1. Markdown 导出。
2. 导出内容模板。
3. 导出记录保存。
4. MVP 样例项目测试。
5. 修复阻断问题。
6. 梳理下一阶段需求。

### 导出内容

第一版导出：

- 项目基础信息。
- 作品核心卡。
- 作品开发案。
- 已确认设定资产。
- 已确认基础关系。
- AI 上下文摘要。

### MVP 验收样例

必须用 3 个不同类型项目测试：

1. 玄幻升级流。
2. 都市悬疑。
3. 科幻群像。

### MVP 完成标准

MVP 完成不是指“没有 bug”，而是指：

- 主链路可以完整跑通。
- 生成内容结构稳定。
- 数据可保存。
- 用户能回到项目继续工作。
- 用户能导出成果。
- 有 3 个样例项目能完整通过。

## 9. Phase 5：中期增强

### 目标

提高产品留存和真实创作价值。

### 推荐功能

1. 关系图谱可视化 v1。
2. 资产版本历史。
3. 设定冲突检测。
4. 隐藏真相 / 剧透保护。
5. Token 预算与上下文裁剪。
6. AI 上下文质量评分。
7. DOCX 导出。
8. 分卷规划。
9. 章节任务草案。
10. 试写启动器。

### 不建议过早做

- 完整工作流引擎。
- 多 Agent 辩论。
- 商业模型评价。
- 联邦治理。

这些功能容易让项目重新膨胀。

## 10. Phase 6：长期 StoryOS 扩展

### 目标

当作品设定工具已经有真实用户后，再考虑扩展为 StoryOS。

### 长期方向

- 试写质量报告。
- 自动修复建议。
- 多平台改写。
- 读者模拟。
- 真实反馈导入。
- 连载管理。
- 创作经验知识库。
- 图驱动 Agent。
- 工作流控制台。

### 长期判断标准

只有当用户真的开始用它管理多个项目，才需要上升到 StoryOS。

否则，长期规划只是好看的负担。

## 12. 周节奏建议

### 每周一：计划

确认本周目标：

- 一个主用户动作。
- 一个核心数据对象。
- 一个可演示页面。
- 一个验收样例。

### 每周三：中点检查

检查：

- 是否能跑起来。
- 是否有接口阻塞。
- 是否有 AI 输出格式问题。
- 是否范围膨胀。

### 每周五：演示与复盘

必须演示真实流程，而不是只汇报“完成了哪些文件”。

复盘问题：

- 用户是否能理解？
- 数据是否保存正确？
- AI 输出是否稳定？
- 下周最小闭环是什么？

## 13. 风险清单

| 风险 | 表现 | 应对 |
|---|---|---|
| 范围膨胀 | 开始做远期治理系统 | 每个功能必须绑定 MVP 主链路 |
| AI 输出不稳定 | JSON 解析失败、字段乱变 | schema 校验、Mock 样例、输出修复 |
| 数据模型过重 | 第一版表太多 | 只建 MVP 必需表 |
| 用户流程复杂 | 页面很多但不知道下一步 | 项目工作台统一承载 |
| 开发陷入基础设施 | 几周没有可演示功能 | 每周必须演示用户动作 |
| 生成内容空泛 | 看起来漂亮但不能保存 | 所有关键内容必须资产化 |

## 15. 项目经理的最终建议

这个项目最有价值的部分，不是让 AI 替作者写完小说，而是让 AI 帮作者把混乱灵感变成可管理的创作资产。

所以第一版要守住三个字：

```text
可保存
```

能保存项目，能保存核心卡，能保存开发案，能保存资产，能保存关系，能保存上下文，能导出成果。

只要这个基础成立，后面的图谱、试写、读者模拟、反馈回流才会有根。
