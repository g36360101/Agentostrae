# AgentOS 项目开发准备计划

## 1. 目的

本计划用于把 AgentOS 从“产品与架构规划完成”推进到“团队可以稳定开发并持续交付”。

准备阶段不扩展产品范围，不提前建设远期能力，只解决四件事：

1. 冻结第一阶段目标和边界。
2. 固化会影响工程结构的基础决策。
3. 建立可运行、可检查、可复现的工程地基。
4. 为第一个用户闭环准备明确的验收样例。

## 2. 当前基线

### 已具备

- 产品定位和目标用户已经明确。
- MVP 主链路和不做事项已经明确。
- Web、API、数据库、AI 层的职责边界已经形成。
- 核心数据实体已有草案。
- AI 建议必须经用户确认后才能进入正史的原则已经确定。
- Web 和小程序分端开发、共享协议和业务类型的方向已经确定。
- 仓库级 AI 开发约束已经建立。
- Web 与小程序已有初步设计资产。

### 尚未具备

- 可安装依赖的 Monorepo 配置。
- 可启动的 Web、API 和数据库工程。
- 可执行的数据模型与首个迁移。
- 稳定的 API 和 AI 输出 Schema。
- 环境变量模板和本地运行说明。
- 统一的 lint、typecheck、test、build 命令。
- CI 基础检查。
- 可复用的 Mock 数据与验收样例。

## 3. 准备阶段的范围

### 本阶段必须完成

- 仓库基线整理。
- 基础技术决策记录。
- pnpm workspace 与 Turborepo 初始化。
- `apps/web`、`apps/api` 基础工程。
- `packages/shared`、`packages/db`、`packages/ai-core` 基础包。
- PostgreSQL 本地开发方案。
- 首批类型、Schema 和 Mock 输出。
- 环境变量模板。
- 统一质量检查命令。
- 最小 CI。
- 三个验收样例的输入数据。
- 第一个纵向闭环的任务拆分。

### 本阶段明确不做

- 真实 AI 模型接入。
- Redis 和 BullMQ。
- 微信小程序业务开发。
- 复杂权限系统。
- 图谱编辑器。
- DOCX / PDF 导出。
- 多 Agent 和远期治理能力。
- 正式生产环境部署。

## 4. 默认技术决策

以下决策作为 Phase 0 默认值。只有出现明确的仓库证据或业务约束时才调整。

| 事项 | 默认决策 | 原因 |
|---|---|---|
| 开发优先级 | Web 优先，小程序后置 | 先验证完整创作工作台闭环 |
| Monorepo | pnpm workspace + Turborepo | 与现有技术路线一致 |
| 主语言 | TypeScript 严格模式 | 前后端和共享协议统一 |
| Web | Next.js + React | 承担完整工作台 |
| API | NestJS | 明确模块边界和后续任务编排入口 |
| 数据库 | PostgreSQL + Prisma | 与数据模型和关系数据需求匹配 |
| 本地数据库 | Docker Compose | 降低环境差异，提供可复现启动方式 |
| 数据校验 | 边界统一使用 Zod；NestJS 适配层复用 Schema | 让 Web、API、AI 输出共享同一套协议 |
| AI | Phase 0 全部使用确定性 Mock | 先验证协议、保存和用户流程 |
| 认证 | Phase 0 使用最小本地认证接口，暂不接第三方登录 | 避免认证集成阻断主链路 |
| UI | 以 `agentos-design` 为视觉参考，以实际组件和 token 为实现源 | 防止设计稿与实现形成两个独立系统 |
| 测试 | 共享规则单测、API 集成测试、首链路最小 E2E | 覆盖最重要的协议和用户动作 |
| CI | install、lint、typecheck、test、build | 为每次变更提供统一底线 |

## 5. 两道开工闸门

### Gate A：允许初始化工程

满足以下条件后，可以开始创建工程骨架：

- [ ] 当前文档和设计改动已经整理，形成清晰的 Git 基线。
- [ ] 第一里程碑被确认，不再加入新功能。
- [ ] 默认技术决策没有阻断性异议。
- [ ] Web 第一批页面和设计参考已经明确。
- [ ] 三个验收样例的主题已经确认。

第一里程碑固定为：

```text
创建项目
  -> 输入创意
  -> Mock 生成至少 3 个高概念
  -> 选择一个高概念
  -> Mock 生成作品核心卡
  -> 保存并再次打开核心卡
```

### Gate B：允许进入业务功能开发

满足以下条件后，Phase 0 完成，可以进入第一里程碑开发：

- [ ] 新开发者只依据 README 可以启动 Web、API 和数据库。
- [ ] Web 可以展示空工作台。
- [ ] API 健康检查可用。
- [ ] Prisma 首个迁移可以在空数据库执行。
- [ ] Web 能通过共享 API Client 调用健康检查。
- [ ] Mock AI 输出能通过共享 Schema 解析。
- [ ] lint、typecheck、test、build 均有统一命令。
- [ ] CI 可以执行基础检查。
- [ ] `.env.example` 不含真实秘密且字段说明完整。
- [ ] 三个样例输入可以被测试或本地开发复用。

## 6. Phase 0 执行顺序

### Step 0：整理仓库基线

目标：让后续工程变更拥有清晰起点。

任务：

- 审查当前未提交文件，区分规划文档、设计资产和临时内容。
- 确认根目录 `AGENTS.md` 纳入仓库。
- 排除 `.DS_Store`、本地缓存、构建产物和秘密文件。
- 在用户明确授权后，形成项目规划基线提交。

验收：

- `git status` 中没有来源不明的文件。
- 后续工程初始化可以单独审查和回滚。

### Step 1：初始化 Monorepo

目标：所有应用和共享包使用统一工具链。

任务：

- 固定 Node.js 和 pnpm 版本。
- 创建 workspace 和 Turborepo 配置。
- 建立统一 TypeScript、ESLint 和格式化配置。
- 定义根级 `dev`、`lint`、`typecheck`、`test`、`build` 命令。

验收：

- 空工程可以完成依赖安装和基础检查。
- 子包不各自维护冲突的工具配置。

### Step 2：建立共享协议包

目标：先稳定数据边界，再开发页面和接口。

首批对象：

- `Project`
- `ProjectIdea`
- `HighConceptCandidate`
- `ProjectCoreCard`
- `AIJob`

同时定义：

- ID、时间字段和状态枚举。
- API 成功响应格式。
- API 错误响应格式。
- 分页格式。
- AI 任务输入和结构化输出格式。

验收：

- 类型和 Zod Schema 同源。
- 合法 Mock 可以解析。
- 缺少必填字段、非法枚举和错误嵌套结构会被拒绝。

### Step 3：建立数据库包

目标：让第一里程碑的数据能够真实保存。

首批表：

- users
- projects
- project_ideas
- high_concept_candidates
- project_core_cards
- ai_jobs

任务：

- 将数据模型草案转为 Prisma schema。
- 建立首个迁移和开发 seed。
- 提供本地 PostgreSQL 启动方式。
- 定义删除、唯一性、外键和时间字段规则。

验收：

- 空数据库可迁移。
- seed 可重复执行或明确说明不可重复部分。
- 核心卡能够追踪来源高概念。
- AI 生成记录能够追踪输入、输出和失败信息。

### Step 4：建立 API 骨架

目标：提供稳定的系统边界和最小可观测性。

任务：

- 创建 NestJS 应用。
- 增加健康检查。
- 建立统一配置读取、错误响应和请求校验。
- 建立 Projects、Ideas、HighConcepts、CoreCards、AIOrchestrator 模块骨架。
- 提供结构化日志，但不记录秘密和完整私人创作内容。

验收：

- API 可以独立启动和优雅退出。
- 健康检查有自动测试。
- 非法请求返回稳定错误结构。

### Step 5：建立 Web 骨架

目标：形成第一里程碑的可见入口。

任务：

- 创建 Next.js 应用。
- 接入 Tailwind CSS、基础 token 和必要的 UI 基础组件。
- 建立应用壳、导航、错误态、加载态和空状态。
- 创建项目列表和项目工作台占位页面。
- 通过共享 API Client 调用健康检查。

验收：

- Web 可以启动并打开工作台。
- API 不可用时显示明确错误，而不是空白页面。
- 页面基础结构与现有设计方向一致。

### Step 6：建立 AI Mock 基础

目标：让产品流程不依赖真实模型也能稳定验收。

任务：

- 定义 `generate_high_concepts` 和 `generate_core_card` 任务。
- 为三个样例提供确定性 Mock 输出。
- 校验所有 Mock 输出。
- 记录 Mock 与真实 Provider 的接口边界。

验收：

- 相同样例得到稳定结果。
- 每个样例至少生成 3 个合格高概念。
- 核心卡包含所有 MVP 必填字段。
- Mock 不绕过真实调用将使用的 Schema 和解析流程。

### Step 7：建立质量与协作底线

目标：让每次提交都能被可靠检查。

任务：

- 添加最小 CI。
- 建立测试目录和命名约定。
- 写清本地启动、迁移、seed 和检查流程。
- 定义 PR 摘要、验证和风险说明方式。
- 建立 ADR 目录或单一决策日志，记录会影响架构的决定。

验收：

- 本地命令和 CI 使用相同检查入口。
- README 中的命令经过实际执行验证。
- 新成员可以在 30 分钟内启动项目。

## 7. 三个验收样例

样例应作为版本化测试资产保存，不只写在文档里。

### 样例 A：玄幻升级流

```text
一个失忆的修仙者发现自己其实是天道写废的草稿。
```

重点验证：世界规则、升级机制、长线谜团和连载潜力。

### 样例 B：都市悬疑

```text
一名失眠的社区医生发现，每位在凌晨三点来诊的病人都会在七天后从城市记录中消失。
```

重点验证：证据链、隐藏真相、剧透标记和阶段性揭示。

### 样例 C：科幻群像

```text
一艘世代飞船即将抵达目的地，船上的五个阶层却分别收到了内容互相矛盾的登陆指令。
```

重点验证：阵营、组织关系、多角色驱动力和核心冲突。

每个样例必须包含：

- 输入创意。
- 3 至 5 个高概念候选。
- 被选择的候选。
- 完整核心卡。
- 预期校验结果。
- 至少一个非法输出样例，用于验证 Schema 拒绝能力。

## 8. 第一里程碑任务切片

Phase 0 完成后，按纵向用户价值切片开发，而不是先做完所有后端再做前端。

### Slice 1：创建并重新打开项目

- 创建项目表单。
- 创建项目 API。
- 项目持久化。
- 项目列表与详情。
- 成功、校验失败和服务失败状态。

### Slice 2：输入创意并生成高概念

- 创意输入。
- 保存原始创意。
- 触发 Mock AI 任务。
- 展示和比较候选。
- 保存 AIJob 与候选来源。

### Slice 3：选择方向并保存核心卡

- 选择候选。
- 生成核心卡。
- 编辑与保存。
- 刷新页面后重新读取。
- 防止多个候选同时被选中。

每个 Slice 都必须包含：

- 用户动作。
- 输入与输出。
- 保存的数据。
- 失败状态。
- 自动化检查。
- 手动验收步骤。

## 9. 安全与成本准备

Phase 0 不接真实模型，但必须提前划清边界：

- 所有秘密只从环境变量读取。
- `.env.example` 只放占位符。
- 日志不输出访问令牌、密码和完整私人创作内容。
- AI Provider 必须位于统一适配层，Web 不得直接调用模型。
- 真实模型启用前必须增加超时、重试上限、token 上限和单任务成本记录。
- AI 建议默认是 `suggested`，不能自动成为 `confirmed`。
- 删除项目和导出数据属于后续高风险功能，需单独设计和验收。

## 10. 主要风险与应对

| 风险 | 早期信号 | 应对 |
|---|---|---|
| 范围继续膨胀 | Phase 0 开始加入图谱、小程序或真实 AI | 以两道 Gate 和第一里程碑为硬边界 |
| Schema 反复变化 | 页面、API、Mock 各自定义字段 | 共享 Schema 先行，变更必须同时检查生产者和消费者 |
| 只搭基础设施没有用户结果 | 一周后仍只有配置文件 | Phase 0 后立即按三个纵向 Slice 交付 |
| Mock 过于理想化 | Mock 绕过解析、错误和 AIJob | Mock 走与真实 Provider 相同的任务和校验入口 |
| 设计与代码分离 | 页面实现出现第二套颜色、间距和术语 | 从现有设计提取最小 token，并指定实现源 |
| 私人内容进入日志 | 调试时打印完整 Prompt 或作品资料 | 默认只记录元数据，敏感正文显式脱敏 |
| 文档命令失真 | README 命令从未在干净环境执行 | Gate B 前逐条运行并记录结果 |

## 11. 准备完成定义

当且仅当以下结果同时成立，项目才算“充分准备好进入开发”：

1. 第一里程碑范围固定且可在一次演示中完整说明。
2. 仓库有清晰基线，工程变更可以独立审查。
3. Web、API、数据库可以通过文档稳定启动。
4. 共享协议、首批数据表和 Mock AI 输出可执行、可校验。
5. 基础质量命令和 CI 可运行。
6. 三个样例能够支撑开发与验收。
7. 安全、日志和 AI 建议状态边界已落实到工程结构。
8. 第一里程碑已拆成可以逐个交付的纵向 Slice。

达到以上条件后，不再继续补启动文档，直接进入 Slice 1。

## 12. 推荐时间安排

| 时间 | 目标 |
|---|---|
| 第 0 天 | 完成 Gate A、整理仓库基线 |
| 第 1 天 | Monorepo、共享工具链、环境模板 |
| 第 2 天 | 共享 Schema、数据库和迁移 |
| 第 3 天 | API、健康检查、统一错误处理 |
| 第 4 天 | Web 应用壳、API Client、基础状态 |
| 第 5 天 | AI Mock、三个样例、测试和 CI |
| 第 6 天 | 从干净环境演练启动流程，修正文档 |
| 第 7 天 | Gate B 验收，进入 Slice 1 |

时间安排是顺序参考，不是为了追求形式上的一周完成。若某个 Gate 条件失败，应修复对应基础问题，不应通过降低验收标准绕过。
