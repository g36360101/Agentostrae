# 项目工作规则

## 核心规则

1. 任何新功能必须能连接到 MVP 主链路。
2. AI 生成内容必须可解析、可保存、可追踪。
3. 用户确认前，AI 建议不能进入 confirmed 状态。
4. 每周都要能演示一个真实用户动作。
5. 不提前建设远期治理系统。

## 决策优先级

当文档之间冲突时，按这个顺序判断：

1. MVP 是否能跑通。
2. 用户是否能理解和使用。
3. 数据是否能保存和追踪。
4. 工程是否简单可维护。
5. 未来是否容易扩展。

## 文件管理规则

### docs

放稳定文档。

### notes

放临时想法、会议记录、未整理脑暴。

### apps

放可运行应用。

### packages

放跨应用共享代码。

### assets

放产品素材、示例图、导出样例。

### scripts

放自动化脚本。

## 原始 172 份文档的使用方式

不要把原始文档当成开发命令。

正确使用方式：

- 当作灵感和参考。
- 优先采纳 P0 / MVP 内容。
- 中期再回看图谱、上下文、读者模拟。
- 远期治理类文件暂时封存。

## 每个功能的验收格式

每个功能写清楚：

```text
用户动作：
输入：
系统行为：
输出：
保存的数据：
失败状态：
验收标准：
```

## 第一个里程碑

完成以下链路即可算第一个里程碑达成：

```text
创建项目 -> 输入创意 -> 生成高概念 -> 选择高概念 -> 生成核心卡 -> 保存
```
